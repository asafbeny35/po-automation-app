# מסמך בדיקות מלא לאפליקציית הניהול PO ב־iPhone

## מטרת המסמך
מסמך זה מגדיר תוכנית בדיקות מלאה, עמוקה ומחייבת לאפליקציית הניהול ל־iPhone, עם דגש מרכזי על שני צירים:

1. איכות עצמאית של האפליקציה:
מהירות, נראות, שימושיות, דיוק נתונים, יישור לימין, ותחושת native אמיתית.

2. השוואת parity מלאה מול מערכת ה־PO הדסקטופית:
כל טאב, כל תת־טאב, כל רכיב, כל כפתור, כל workflow, כל מסלול יצירה, כל מסלול שליחה, כל מסלול פרסור, וכל מסלול סנכרון שקיימים ב־`http://localhost:8000/` חייבים להיבדק מול גרסת האייפון.

היעד אינו "כיסוי טוב". היעד הוא כיסוי מקסימלי ומבוקר, ברמה שמדמה עבודת QA מצטברת של צוות בדיקות גדול לאורך זמן, כולל:
- בדיקות אוטומטיות
- בדיקות exploratory ידניות
- בדיקות רגרסיה
- בדיקות עומס שימוש
- בדיקות UI/UX
- בדיקות אינטגרציה
- בדיקות parity מול הדסקטופ

## מתודולוגיית הבדיקה

### עקרון מנחה
כל בדיקה באייפון נשאלת בשתי רמות:

1. האם הפיצ'ר עצמו עובד היטב.
2. האם הוא קיים ומתנהג כמו מקבילו בדסקטופ, או שיש פער מוצהר ומתועד.

### סוגי בדיקות מחייבים
- Smoke
- Sanity
- Functional
- Integration
- Regression
- UX review
- Visual review
- RTL review
- Data completeness review
- Exploratory
- Error handling
- Navigation consistency
- Session/Auth lifecycle
- Offline/timeout degradation where applicable

### רמות חומרה
- `P0`: חסימה מלאה של כניסה, ניווט, טעינת מודול, יצירה, שליחה, או אובדן נתונים
- `P1`: פער קריטי בפיצ'ר ליבה או פער parity משמעותי מול הדסקטופ
- `P2`: בעיית UX/UI משמעותית, RTL, נראות, או מידע חסר שפוגע בשימוש שוטף
- `P3`: ליטוש, מיקרו־אינטראקציה, ניסוח, או חוסר עקביות לא חוסם

### קריטריון יציאה
לא מכריזים שהאפליקציה תקינה עד שכל אלו מתקיימים:
- אין `P0`
- אין `P1` פתוחים
- כל פער parity מול הדסקטופ מתועד
- כל מסך מרכזי עבר smoke
- כל flow קריטי עבר E2E
- כל מסך עבר בדיקת RTL בסיסית לפחות
- אין קטעי טקסט חתוכים, כפתורים מתים, או ניווט שנתפס כ־web-like במקום native

# 1. מהירות טעינה

## 1.1 מטרות ביצועים
- מסך פתיחה ראשון: זמן עד UI אינטראקטיבי
- זמן כניסה אחרי אימות
- זמן מעבר בין טאבים ראשיים
- זמן מעבר בין תתי־טאבים
- זמן טעינת רשימות ארוכות
- זמן חזרה ממסך detail למסך רשימה
- זמן פתיחת מודאלים
- זמן ביצוע פעולה עם spinner/progress
- זמן רענון נתונים

## 1.2 מדדי קבלה
- מעבר בין טאבים ראשיים:
  - לא אמור להציג full-screen loading בכל מעבר שגרתי
  - תגובת UI מיידית, עד `300ms` לחיווי ראשוני
- תת־טאב פנימי:
  - מעבר מיידי עם skeleton/placeholder מקומי אם צריך
  - לא full-screen splash
- טעינת מסך כניסה:
  - מסך טעינה מלא על כל המסך
  - לא כרטיס קטן במרכז
- כניסה עם Face ID:
  - זמן תגובה קצר, חיווי ברור, fallback עובד
- רשימות:
  - גלילה רציפה
  - בלי קפיצות layout
  - בלי re-render אגרסיבי

## 1.3 תרחישי בדיקה

### 1.3.1 טעינת אפליקציה קרה
- פתיחת האפליקציה כשהיא סגורה לגמרי
- בדיקת זמן עד הופעת loading
- בדיקת זמן עד מסך כניסה / מסך נעילה / מסך תוכן
- בדיקת fullscreen אמיתי
- בדיקת animation סביר

### 1.3.2 טעינת אפליקציה חמה
- סגירה לרקע והחזרה
- נעילה/פתיחה של המסך
- בדיקת האם יש טעינה מיותרת מחדש

### 1.3.3 כניסה דרך קוד למייל
- שלח קוד
- אמת קוד
- מדוד זמן בין אימות לבין מעבר למסך ראשון

### 1.3.4 כניסה דרך Face ID
- בדיקת auto-prompt
- בדיקת unlock חוזר אחרי lock
- בדיקת כישלון ביומטרי וחזרה למסלול אחר

### 1.3.5 מעבר בין טאבים
- מרכז -> הזמנות
- הזמנות -> תשלומים
- תשלומים -> כספים
- כספים -> לקוחות
- לקוחות -> מרכז
- לוודא שאין full-screen loader מיותר

### 1.3.6 מעבר בין תתי־טאבים
- הזמנות: בעבודה / היסטוריה / הצעות / אישורי מסירה
- תשלומים: הכל / לגבייה / לתשלום
- כספים: חשבוניות / ניכויים / בנק / דוחות
- לקוחות: חיפוש / רשימות / פרטים
- מרכז: כרטיסים / קיצורי דרך / מודולים

### 1.3.7 טעינת רשימות כבדות
- לקוחות
- היסטוריית הזמנות
- היסטוריית חשבוניות
- תשלומים והעברות
- מנהלי פרויקטים
- שיווק

## 1.4 כלי אוטומציה
- מדידת timestamps ב־launch וב־onAppear של מסכים מרכזיים
- UI automation למדידת זמן בין tap ל־first frame usable
- לוג פר מדידה:
  - screen
  - source state
  - cold/warm
  - API used
  - time to visible
  - time to interactive

## 1.5 נקודות השוואה לדסקטופ
- אם בדסקטופ מעבר בין תתי־טאבים מיידי, האייפון לא אמור להחמיר התנהגות
- אם בדסקטופ מציג spinner מקומי, האייפון לא אמור להציג splash מלא
- פעולות רענון צריכות להרגיש קלות ומהירות יותר באייפון, לא איטיות יותר

# 4. מידע חסר

## 4.1 מה נחשב "מידע חסר"
- שדה שקיים בדסקטופ ואינו מופיע באייפון
- שדה שקיים ברשומה אך לא מוצג למשתמש
- metadata מוסתר שצריך לקבל החלטה לפיו
- קישור/קובץ/אסמכתא/סטטוס שלא מוצגים
- empty state שאינו מסביר מה חסר

# 5. יישור לימין

## 5.1 עקרון בסיס
האפליקציה היא אפליקציה עברית. ברירת המחדל היא RTL מלאה.

כל רכיב שמיושר לשמאל בלי סיבה מוצדקת הוא באג, אלא אם מדובר בנתון לועזי/מספרי שהפורמט שלו מחייב חריגה.

## Definition of Done
האפליקציה לא תיחשב תקינה עד שיתקיימו כל התנאים:
- הניווט מרגיש native ולא web-like
- מסך טעינה fullscreen ומעוצב היטב
- אין פערי RTL פתוחים במסכים קריטיים
- אין כפתורים מתים במסכי ליבה
- `+ יצירה` בהזמנות עובד
- קיצורי דרך במרכז עובדים
- כל מעבר בין תתי־טאבים הוא מיידי וסביר
- אישורי מסירה מוצגים נכון, קומפקטי, ועם מידע מלא
- כל פער parity מול הדסקטופ זוהה, תוקן או תועד מפורשות
- כל flow קריטי עבר בדיקה חוזרת אחרי התיקונים
