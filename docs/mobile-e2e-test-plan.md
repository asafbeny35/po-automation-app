# BenYacovMobile E2E Test Plan

תאריך: 2026-06-19

מטרה: להגדיר מסמך בדיקות מקיף, מלא ואופרטיבי לאפליקציית `BenYacovMobile`, ברמת end-to-end אמיתית לכל flow משמעותי במערכת.

העיקרון המנחה:
- כל בדיקה נכתבת סביב תוצאה עסקית מלאה, לא סביב כפתור בודד.
- כל flow נבדק מתחילתו ועד סופו: קלט, פרסינג, שמירה, סנכרון, יצירת מסמכים, פתיחת קבצים, שליחה, ועדכון state.
- בכל מקום שיש תלות ב־backend, Google Drive, Gmail OAuth, GreenInvoice, Supabase או קבצים מקומיים, הבדיקה חייבת לאמת גם את שכבת האינטגרציה ולא רק את ה־UI.

### 1.1 רשימת parity מחייבת מול הדסקטופ

המסמך לא מסתפק בכיסוי "מסכים ראשיים", אלא מחייב parity מול כל היכולות הקיימות בפועל ב־PO הדסקטופי.

טאבים ראשיים שמחויבים להיבדק:

1. הזמנות
2. הכנסות
3. תשלומים והעברות
4. תמחור ועצי מוצר
5. מלאי
6. מנהלה
7. כספים
8. משרד
9. לקוחות
10. מנהלי פרויקטים
11. שיווק

שרשראות עסקיות מחייבות parity מלא:

1. העלאת הזמנת רכש -> פרסור -> תיקון ידני -> יצירת מסמכים -> שמירה -> סנכרון -> שליחה
2. יצירה ידנית -> טעינה למסך -> חישוב סכומים -> יצירת מסמכים -> היסטוריה
3. אישור מסירה -> העלאת קובץ -> תצוגה -> שליחה -> עדכון סטטוס
4. חשבונית ספק -> העלאה -> OCR/פרסור -> תיקון -> שמירה -> שיוך דיווח -> ייצוא/שליחה
5. הצעת מחיר שיווקית -> ייצוא אקסל -> ייבוא שינויים -> יצירת הצעה עדכנית -> שליחה
6. תעודת משלוח ספק -> פרסור -> שמירה -> הוספה למלאי -> עדכון כמויות
7. חומרי שכר -> שעות/אישורי מחלה -> הכנה -> תצוגה מקדימה -> שליחה לרו"ח
8. refresh תפעולי מול GreenInvoice / Gmail / Google Drive / Supabase

## 2. הנחות ותנאי סף

לפני הרצת הסבב:

1. השרת של `po_automation_app` זמין ועובד.
2. `BenYacovMobile` מחובר ל־backend הפעיל ולא ל־localhost-only fake flow.
3. הרשאות Gmail OAuth ו־Google Drive OAuth בתוקף.
4. חיבורי GreenInvoice PROD ו־Sandbox תקינים.
5. בסיס הנתונים של Supabase זמין.
6. יש חשבון בדיקה פעיל עם הרשאות מלאות.
7. על הסימולטור או המכשיר יש קבצי fixture זמינים ל־PDF, JPG, PNG, XLSX, EML.
8. Face ID / device authentication זמין על הסימולטור או המכשיר, אם נבדקת כניסה ביומטרית.

## 3. סט נתוני בדיקה

4. קובץ הזמנת רכש עם פרטי לקוח חלקיים.

6. מספר חשבוניות ספק בפורמטים שונים: PDF רגיל, PDF סרוק, JPG כפול, EML.
7. קובץ XLSX של שעות נוכחות.
8. קובץ supporting HR, כגון אישור מחלה.
9. קובץ תעודת ספק.
10. קובץ מסמכי שיווק.

## 4. אסטרטגיית הרצה

כל test case יסווג כך:

- `AUTO`: מתאים לאוטומציה מלאה.
- `SEMI`: דורש נתון חיצוני או בדיקת מסמך/שירות חיצוני.
- `MANUAL`: דורש בדיקה אנושית של output עסקי, נראות, או פעולה במערכת חיצונית.

בכל מקרה:

1. יש להריץ קודם smoke רחב.
2. אחריו סבב E2E מלא.
3. אחריו regression לפי המודולים שהשתנו.
4. אחריו סבב RTL/UI/UX.

## 20. מטריצת parity מול הדסקטופ

לכל release של המובייל חובה לבצע checklist הבא:

1. לעבור על כל endpoint תפעולי רלוונטי בדסקטופ.
2. לוודא שיש לו לפחות נתיב native אחד במובייל:
   - מסך ייעודי
   - sheet ייעודי
   - או `NativeOperationsHubView`
3. לוודא שהמובייל לא רק "פותח" מסך, אלא משלים את ה־flow העסקי:
   - קליטת קלט
   - שליחה לשרת
   - קבלת תשובה
   - עדכון state
   - חיווי הצלחה/שגיאה
4. לוודא שהמסמך/קובץ/שליחה נוצרו בפועל גם במערכת היעד.

### 20.1 כיסוי מחייב לפי טאבים

| טאב דסקטופ | מה חייב להיות מכוסה במובייל |
| --- | --- |
| הזמנות | פרסור PDF, יצירה ידנית, טעינה למסך, חישוב סכומים, sandbox/prod finalize, merged docs, היסטוריה, הצעות, עבודה, אישורי מסירה, שליחה |
| הכנסות | לקוחות הכנסות, upcoming payments, יצירת קבלה, תזכורות בוואטסאפ/מייל |
| תשלומים והעברות | טבלאות `הכל` / `לגבייה` / `לתשלום`, יצירה, עדכון, מחיקה, סימון שולם, רענון |
| תמחור ועצי מוצר | טעינת עצי מוצר, עריכה, שמירה, חישובי תמחור רלוונטיים |
| מלאי | מצב מלאי, תעודות ספק, הזמנות רכש, שליחת רכש, שיקום מלאי, ניכויי sandbox |
| מנהלה | Pazomat, Sibus, מסמכי עסק, תיקיות Drive, מסמכי ספק/לקוח/ISO |
| כספים | העלאות חשבוניות, OCR, שמירה, שיוך דיווח, יצוא, שליחה, מחיקה, due dates, ניכויים, הגדרות |
| משרד | מחשבונים, נתוני משרד, רשתות, משתמשים, פתיחת קישורים/מסמכים |
| לקוחות | חיפוש, רענון, יצירה, עדכון, שיוך תחום, מחיקה, inactive, שליחת מיילים, מסמכים |
| מנהלי פרויקטים | טעינה, העלאה/פרסור, עדכון טבלאות ופרטי קשר |
| שיווק | pipeline, reminders, notes, מסמכים, construction companies, work managers, email/whatsapp, quote update flow |

### 20.2 כיסוי מחייב לפי endpoints תפעוליים

מינימום נדרש:

1. כל route קריטי שמוגדר במערכת הדסקטופ חייב להיות ממופה ל־test case אחד לפחות.
2. כל route mutating חייב להיבדק גם על success וגם על failure.
3. routes קריטיים במיוחד שחייבים להיות עם E2E מלא:

### 20.3 התאמה לסוויטת הבדיקות הקיימת

1. API contracts:
   - `tests_full_system/api`

   - `tests_full_system/e2e`
3. Mobile native E2E:
   - XCUITest / UI automation מקביל, לפי אותם flow IDs

כל flow חדש במובייל חייב להצביע על:

1. flow עסקי במסמך זה
2. route/endpoint תפעולי
3. source of truth בדסקטופ
4. תסריט אוטומטי או חוב בדיקות פתוח
